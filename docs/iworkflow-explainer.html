How it works

Code dispatches · CLIs execute · the journal remembers

You describe a workflow as data (a JSON spec) or pick a named recipe. A deterministic Python runner fans the steps across the subscription CLIs with a per-provider concurrency cap, fails over to another subscription when one is throttled, and writes every step to a durable journal so a crash or a rate-limit window can resume exactly where it stopped.

🎯 Provider asymmetry

Codex — structured doers (native output-schema). Gemini — schema-less adversarial audit & 1M-token sweeps. Claude — interactive driver / delicate cores (scarce, cap 1).

🔁 Failover & resume

Throttled subscription? Re-dispatch to the next, persist a cooldown, and recover completed steps from the ledger across rate-limit windows.

🔒 Data, not code

A spec is JSON — safe to accept from an agent over MCP. No closures, bounded by an explicit safety policy.

Five step kinds compose any shape

🧵 Templating threads results

Prompts render against the run context:

{{params.*}} {{steps.id.value.*}} {{loop.collected}} {{item}} {{prev}} {{supervisor.steps}}

A string that is exactly one {{token}} keeps the raw object (a list stays a list); otherwise it stringifies inline.

⏹️ Six ways a loop stops

times N rounds count ≥ N items dry K empty rounds budget N tokens agent critic says STOP vote majority of judges

The agent/vote loop is the “completeness critic”: it runs until a judge is satisfied, not until a counter trips — under a hard cap.

A real spec

Gate → fan two reviews → independent audit

This is the built-in review recipe. A gate blocks bad input before spending fan-out work; two disjoint reviews run in parallel; then an independent auditor that deliberately ignores the reviewers.

spec.json
"steps": [
  // 1 · gate — abort the whole run if the design is fundamentally broken
  { "id":"gate", "kind":"agent", "schema":"gate", "prefer":["claude","codex"],
    "gate":{ "field":"verdict", "abort_on":"BLOCKED" },
    "prompt":"Is {{params.topic}} sound enough to review? verdict=DONE or BLOCKED." },

  // 2 · fan — two disjoint reviews, concurrently (a barrier)
  { "id":"fan", "kind":"parallel", "needs":["gate"], "agents":[
      { "id":"a", "schema":"review", "prefer":["codex","gemini"],
        "prompt":"Review for CORRECTNESS. verdict=PASS/ISSUES.\n{{params.subject_a}}" },
      { "id":"b", "schema":"review", "prefer":["gemini","codex"],
        "prompt":"Review for ROBUSTNESS. verdict=PASS/ISSUES.\n{{params.subject_b}}" } ] },

  // 3 · audit — independent adversary; trusts NO reviewer
  { "id":"audit", "kind":"agent", "needs":["fan"], "prefer":["gemini","codex"],
    "prompt":"Independent auditor. Is there a real correctness hole across these?" } ]
The output bundle is always { status, name, output, steps:{ id → value }, aborted_at? } — the rendered output plus every step's structured value, ready to thread on.
The differentiator

The supervisor — adapt the plan mid-run, without losing determinism

Most engines run a frozen graph. iworkflow adds a coordinator that reads the accumulated state at a checkpoint and reshapes the remaining plan — the safe, data-not-code answer to “an orchestrator that programs the next sub-plan”. It emits a validated decision; the deterministic executor applies it.

The decision (as data)

{ "action": "continue | adjust | abort",
  "skip":   ["future_step"],   // drop work
  "set_params": { "k": "v" }, // re-aim
  "inject": [ { /* a new step */ } ] }

Invariants that keep it safe

• Only the tail is mutable — the past is immutable.
• Injected steps can’t escalate sandbox/tools (same validator); a bad inject degrades gracefully.
• Bounded by max_supervisions.
• Journaled → a resume re-applies the same mutation with zero new calls.

Fire only on deviation. An optional when guard — a tiny declarative predicate over the same state ({"path":"steps.fan.value","select":"value.verdict","in":["ISSUES"]}, combinable with all/any/not) — lets the coordinator run only when something actually drifts. The on-track path spends zero coordinator tokens. The adaptive_review recipe uses it to inject a deep audit only when a review flags issues.

Host projects drop their own *.json specs into .iworkflow/recipes/ and they appear alongside the built-ins — iworkflow stays domain-agnostic. A dynamic spec that proves itself gets saved as a recipe: the dynamic → confirmed → preset calcification.

How to run

Three front doors — same engine, for humans and agents

It’s not “humans use the CLI, agents use MCP”. The CLI, the MCP tool, and the Python SDK all call the same engine — the choice is how you reach it, not what it can do. An agent can drive iworkflow over either the CLI (shell out via Bash) or MCP (a registered tool).

① CLI — a person, CI, or an agent via Bash

# list recipes + their params
iworkflow workflows

# run a named recipe
iworkflow run review \
  --params '{"topic":"the scheduler",
             "subject_a":"…","subject_b":"…"}'

# run your own spec, resumable by id
iworkflow run --spec spec.json --run-id job1

# telemetry from past runs
iworkflow stats --run-id job1

② MCP — an agent as a registered tool

Register once, then any agent in the repo gets three tools:

iworkflow_ping iworkflow_list_workflows iworkflow_workflow

iworkflow register # wire into .mcp.json / .codex

# then the agent calls, e.g.:
iworkflow_workflow(
  workflow="adaptive_review",
  params={…})
# …or spec={…} to define its own,
# …or goal="…" for a quick fan→synth.
Same engine, different reach. An agent can use both: CLI (Bash) gives the full toolbelt (stats, catalog, --run-id resume) but you parse stdout; MCP returns a structured result and is discoverable in the tool list. Prefer MCP when it’s registered; fall back to CLI when you need stats/catalog or there’s no MCP host (CI, a cron-agent, a script). A third door — the Python SDK (run_spec(...)) — is the only one that can widen Limits or swap providers, and is what tests use. The iworkflow_workflow tool itself drives three ways: spec= (your own), workflow=+ params= (a recipe), or goal= (a quick fan→synthesize).
Safety & telemetry

Bounded by policy · observable for free

🛡️ A spec can arrive from an untrusted agent

So every run is bounded by a conservative Limits policy: only a read-only sandbox (a spec can’t request a privileged one), no tool injection, and caps on total agent calls, parallel width, pipeline items, loop depth/iterations, and supervisor mutations — no spec can fork-bomb paid CLI spawns. Trusted CLI/SDK callers widen it explicitly.

📈 Every run logs itself

Under .iworkflow/runs/<id>/: events.jsonl (full trace), ledger.jsonl (durable per-agent record — provider, attempts, tokens, cost, latency; drives resume), and step checkpoints. Explore with iworkflow stats or provider_stats() / run_summary(). Re-running the same id spends zero new tokens on completed steps.
