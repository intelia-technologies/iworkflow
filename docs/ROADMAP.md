# iworkflow roadmap

_Drafted by a real iworkflow run (roadmap-1): Codex + Gemini proposed, codex synthesized._

| Rank | Title | Why | Effort |
|---:|---|---|---|
| 1 | Durable run ledger + throttle-aware resume loop | This is the core reliability foundation: record prompt/schema fingerprints, attempts, provider, latency, status, retry windows, errors, and output hashes so throttled runs pause/fail over/resume deterministically instead of redispatching blindly. | M |
| 2 | Subscription-free fault-injection test suite | Promote the demos into real CI coverage for caps, failover, corrupt journals, stale resume rejection, schema fallback, timeouts, and rate-limit banners before expanding the public surface. | M |
| 3 | MCP `workflow(script)` gateway | Make iworkflow usable directly from Codex, Gemini, and Claude by exposing a thin optional MCP/stdin entrypoint with status streaming, structured results, and journal-backed resume. | M |
| 4 | Worktree-backed safe parallel writes | Real code-writing fan-out needs per-agent git worktrees, patch/artifact capture, deterministic merge/review, and cleanup so workers cannot corrupt the shared checkout. | L |
| 5 | Journal-driven UX + empirical routing | Once the ledger exists, use it for progress display, throttle visibility, run history, and measured provider-routing recommendations instead of static priors. | M |

I’d do them in this order. The ledger comes first because it strengthens iworkflow’s main promise: deterministic orchestration across subscription-limited CLIs. MCP is important for adoption, but it should sit on top of reliable resume/throttle semantics rather than expose the current spike behavior.
